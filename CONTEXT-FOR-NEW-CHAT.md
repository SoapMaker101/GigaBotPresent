## 1. Что такое GigaBotPresent

- **Папка:** `GigaBotPresent` — отдельный «проект» рядом с основным репозиторием gigabot (или скопированный в другой путь). Всё для презентации CSM в Сбере лежит внутри неё.
- **Цель презентации:** показать CSM, как LLM встраивается в контур (аналогия: Cloud.ru + GigaChat), продемонстрировать RAG и задачи, дать язык метрик и снять «туман» перед GigaEnterprise. Итог: CSM понимают, как работает LLM в контуре, и при разговоре/планировании с клиентом знают, как им помогут продукты Сбера (в т.ч. GigaEnterprise).
- **Демо:** один бот (форк GigaBot) в Telegram или CLI; сценарий на 15–25 мин: вступление → база знаний + RAG → задачи + приоритизация → опционально аналитика → метрики → GigaEnterprise.

## 3. Что нужно сделать в новом чате

### 3.1 Проверка кода по документации

- Пройти по **сценарию демо** из `docs/06-Scenarij-demo-CSM.md` и убедиться, что каждый шаг реализуем текущим кодом:
  - Блок 2: создание проекта (project), база знаний (knowledge: create_project, index_folder), поиск (knowledge search). Пути, имена проектов, папка «Документация» — соответствие с doc.
  - Блок 3: создание задач (tasks add с priority, deadline), список (tasks list), запрос «подготовь отчёт по задачам: что выполнить первым по критичности, сложности и дедлайну» — должен давать отчёт (навык task_report + tasks(list)).
  - Блок 4 (опционально): «сводка» / «что загружено, какие задачи, что в базах» — tool summary.
- Сверить реализацию с **ТЗ** `tech-spec/TZ-Fork-GigaBot-Present.md`: критичные функции (RAG, tasks, cron, project, file, memory, context, skills) на месте; summary и отчёт по задачам соответствуют п. 3.1 и 3.2 ТЗ.
- Убедиться, что **описание инструментов** в системном промпте (`agent/context.py`) не противоречит списку зарегистрированных tools и сценарию (в т.ч. summary и приоритизация).

### 3.2 Порезать лишние tools и адаптировать под сценарий

По ТЗ для демо **критичны** только:

- **project** — создание проектов, папки, move_file, send_files (для «положи файлы в проект», «создай проект»).
- **file** — read, write, list (для чтения файлов при индексации, создания файлов при необходимости).
- **knowledge** (RAG) — create_project, list_projects, index_file, index_folder, search.
- **tasks** — add, list, update, complete, remove (приоритет, дедлайн).
- **cron** — add, list, remove (напоминания по сценарию).
- **message** — отправка ответов и медиа пользователю.
- **summary** — сводка по проектам, задачам, RAG (для блока «аналитика»).

**Не нужны для сценария демо (по документу 06 и ТЗ):**

- **exec** — выполнение shell-команд (риск, не используется в демо).
- **web** — поиск в интернете (в сценарии не заложен).
- **spawn** — подагенты (в демо не используются).
- **ocr** — распознавание текста с фото (в сценарии не заложен).
- **generate_image** (Kandinsky) — генерация картинок (в сценарии не заложен).
- **voice_note** (SaluteSpeech) — TTS/STT (в сценарии не заложен).

**Задача:**  
- В форке `GigaBotPresent/gigabot/` оставить только перечисленные критичные tools (или включить отключение «лишних» по конфигу/флагу «presentation», как в п. 3.3 ТЗ).  
- Убрать регистрацию exec, web, spawn, ocr, kandinsky, salute_speech из `agent/loop.py` (или сделать условной: например, при `presentation_mode=True` или конфиг `agent.presentation_mode` регистрировать только критичный набор).  
- Привести системный промпт в `agent/context.py` в соответствие с фактическим списком tools (удалить или не упоминать вырезанные инструменты в identity/правилах).  
- Проверить, что ни один критичный сценарий из документа 06 не сломан (создание проекта, RAG, задачи, отчёт, сводка, напоминания).

### 3.3 Дополнительно (по возможности)

- Режим «presentation»: конфиг или env (например `GIGABOT_PRESENTATION_MODE=true` или в `agent` секции конфига `presentationMode: true`), при котором в loop регистрируется только минимальный набор tools для демо — описать в ТЗ и при необходимости реализовать в `config/schema.py` и `loop.py`.
- Убедиться, что SubagentManager и другие зависимости от вырезанных tools не ломают запуск (например, spawn не регистрируется — подагенты не используются; остальная логика loop не должна падать).

## 4. Где что лежит (пути относительно GigaBotPresent)

| Что | Путь |
|-----|------|
| Документация презентации | `docs/01-...md` … `docs/07-...md` |
| Сценарий демо (тайминг, фразы) | `docs/06-Scenarij-demo-CSM.md` |
| Техзадание на форк | `tech-spec/TZ-Fork-GigaBot-Present.md` |
| Инструкция по демо-пресету | `DEMO-PRESET.md` |
| Код форка (пакет) | `gigabot/gigabot/` |
| Регистрация tools | `gigabot/gigabot/agent/loop.py` — метод `_register_default_tools` |
| Системный промпт, список инструментов в identity | `gigabot/gigabot/agent/context.py` |
| Tool summary | `gigabot/gigabot/agent/tools/summary.py` |
| Навык отчёта по задачам | `gigabot/gigabot/skills/task_report/SKILL.md` |
| RAG, tasks, cron, file, project | `gigabot/gigabot/agent/tools/` (rag.py, tasks.py, cron.py, filesystem.py) |
| Конфиг (схема) | `gigabot/gigabot/config/schema.py` |

Рабочая директория для команд и правок — корень репозитория, где лежит папка `GigaBotPresent`, либо сама `GigaBotPresent` (в зависимости от того, откуда запускается чат). Код форка — всегда `GigaBotPresent/gigabot/`.
