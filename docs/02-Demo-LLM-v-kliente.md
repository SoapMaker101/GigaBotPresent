# Тема 2: Демонстрация работы LLM в клиенте

**Для презентации CSM в Сбере.** Цель — показать на примере GigaBot: работа по запросу, анализ информации (RAG), агентные сценарии (задачи, приоритизация).

---

## База знаний: создание, настройка поиска, что такое RAG

### Создание базы знаний

- **Действие пользователя:** «Создай базу знаний для проекта „Демо-проект“ и добавь в неё всё из папки Документация.»
- **Ожидание:** Вызовы `knowledge(create_project, project="Демо-проект")` и `knowledge(index_folder, project="Демо-проект", folder_name="Документация")`. Файлы из папки проекта (PDF, DOCX, TXT и т.д.) читаются, разбиваются на чанки, эмбеддинги считаются через GigaChat (EmbeddingsGigaR), чанки сохраняются в ChromaDB в коллекции, привязанной к проекту.

### Настройка поиска

- Поиск «настраивается» через выбор проекта и формулировку запроса. Отдельной «настройки» в UI нет — агент вызывает `knowledge(search, project="...", query="...")`.
- При необходимости можно показать параметр `top_k` (сколько фрагментов возвращать).

### Что происходит при RAG (Retrieval-Augmented Generation)

1. Пользователь задаёт вопрос по документам (например: «Где в документации говорится про сроки поставки?»).
2. Агент вызывает `knowledge(search, project="Демо-проект", query="сроки поставки")`.
3. Текст запроса преобразуется в вектор эмбеддингов (GigaChat Embeddings), в ChromaDB выполняется семантический поиск (cosine similarity), возвращаются наиболее релевантные чанки с метаданными (источник, релевантность).
4. Эти чанки подставляются в контекст ответа LLM; модель формулирует ответ на основе найденных фрагментов, а не «из головы».

Для демо достаточно показать: создание проекта RAG → индексация папки → вопрос по содержимому → ответ с опорой на документы.

## Задачи по времени: приоритизация и отчёт

### Базовые возможности (есть в GigaBot)

- **Создание задачи:** «Добавь задачу: подготовить отчёт до пятницы, приоритет высокий» → `tasks(add, title="...", deadline="...", priority="high")`.
- **Список задач:** «Покажи мои задачи» / «Задачи по проекту X» → `tasks(list)` или `tasks(list, project="X")`. В выводе — приоритет (low/medium/high), дедлайн, статус (todo/in_progress/done).

### Сценарий «подготовить отчёт по критичности, сложности и дедлайну»

- **Запрос пользователя:** «У меня ряд задач по времени. Подготовь отчёт: какие выполнить первыми, какие позже — по фактору критичности, сложности и дедлайну.»
- **Ожидание:** Агент вызывает `tasks(list)` (при необходимости с фильтрами), получает список с полями `priority`, `deadline`, `status`, `project`. На основе этих данных формирует текстовый отчёт: группировка по срочности/дедлайну, по приоритету (high/medium/low); при наличии в форке полей «сложность» или «критичность» — включить их в отчёт (см. ТЗ форка).

Для демо важно показать, что LLM не просто перечисляет задачи, а **анализирует и приоритизирует** их по заданным критериям (в пределах доступных полей).

## Пошаговые сценарии для демо

| Шаг | Действие пользователя | Ожидаемые вызовы tools |
|-----|------------------------|------------------------|
| 1 | «Создай проект „Презентация CSM“.» | `project(create, name="Презентация CSM")` |
| 2 | «Создай базу знаний „Презентация CSM“ и добавь в неё папку Документация.» | `knowledge(create_project, project="Презентация CSM")`, `knowledge(index_folder, project="Презентация CSM", folder_name="Документация")` |
| 3 | «Найди в базе знаний всё про метрики внедрения.» | `knowledge(search, project="Презентация CSM", query="метрики внедрения")` |
| 4 | «Добавь задачу: подготовить слайды до завтра 18:00, высокий приоритет.» | `tasks(add, title="подготовить слайды", deadline="...", priority="high")` |
| 5 | «Подготовь отчёт по задачам: что сделать первым по критичности и дедлайну.» | `tasks(list)` + текстовый отчёт от агента |

При необходимости перед демо создать тестовый проект с папкой Документация и 1–2 файлами, а также 3–5 задач с разными приоритетами и дедлайнами.
