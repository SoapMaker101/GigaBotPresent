# Анализ слайдов: 10 признаков хорошего/плохого кейса и примитивы LLM

**Для презентации CSM в Сбере.** Документ проверяет корректность слайдов и связывает их с контекстом GigaBot и внедрения LLM.

---

## 10 признаков хорошего кейса — связь с GigaBot

| № | Признак | Корректность | Связь с GigaBot / внедрением LLM |
|---|---------|--------------|-----------------------------------|
| 1 | Объём и рутина: много повторяющихся операций чтения/написания/сравнения текста, форм, тикетов | Корректно | RAG (поиск по документам), задачи, напоминания — как раз рутина; индексация папок, ответы по базе знаний. |
| 2 | Разнобой форматов: PDF/сканы, фото, полуструктур, данные «на полях» | Корректно | Поддержка PDF, DOCX, XLSX, TXT; OCR с фото; в RAG — единый семантический поиск поверх форматов. |
| 3 | Разрыв «человек–система»: людям надо глазами читать/сопоставлять, потом вбивать в CRM/ERP | Корректно | Агент читает документы, ищет в RAG, формирует ответы; диалог снижает разрыв (запрос → ответ без ручного ввода в систему в демо). |
| 4 | Исключения и «длинный хвост»: много нестандартных кейсов, не покрытых жёсткими правилами | Корректно | LLM обрабатывает свободные формулировки; RAG подтягивает релевантные фрагменты при нестандартных вопросах. |
| 5 | Мультиканальность: e-mail, чат, звонки, мессенджеры, соцсети | Частично | В GigaBot — один канал (Telegram); для кейса мультиканальность — про архитектуру GigaEnterprise (несколько каналов в одном агенте). |
| 6 | Мультиязычность: стык данных на нескольких языках, ВЭД | Частично | GigaChat и RAG работают с русским; мультиязычность — общий признак хорошего кейса для LLM, не специфика текущего демо. |
| 7 | SLA / время-до-ответа важно: контакт-центры, закупки, поддержка, риск-оценка | Корректно | Быстрый ответ за счёт RAG и агента; для CSM — операционная метрика «сокращение времени процессов». |
| 8 | Наличие «центров обслуживания»: контакт-центры, бек-офисы, экспедиции документов | Корректно | GigaBot демонстрирует прототип такого центра (один агент с базой знаний и задачами); в клиенте — масштаб до КЦ/бек-офиса. |
| 9 | Текучесть персонала: много внутренних регламентов, знаний «в головах» | Корректно | База знаний (RAG) фиксирует знания в документах; поиск по ним не зависит от конкретного сотрудника. |
| 10 | Сезонные пики / «дни X»: всплески обращений или документооборота | Корректно | Агент масштабируется без пропорционального роста FTE; для клиента — аргумент про avoided costs и FTE. |

**Вывод:** слайд «10 признаков хорошего кейса» корректен; с GigaBot напрямую связаны признаки 1–4, 7–10; 5–6 — общий контекст LLM-кейсов.

## Примитивы LLM — маппинг на инструменты GigaBot

| Примитив | Описание | Инструмент / сценарий в GigaBot |
|----------|----------|----------------------------------|
| 1. Классификация | Маршрутизация тикетов, темы обращений, intent, риск-метки | Модель по тексту запроса выбирает действие (RAG, задачи, файлы и т.д.); явного отдельного tool нет. |
| 2. Извлечение сущностей | Заявки, анкеты, счета, контракты, e-mail-заказы | OCR (текст с фото); чтение файлов (file read); при индексации в RAG — текст из PDF/DOCX. |
| 3. Семантический поиск | База знаний, регламенты, каталоги, подбор аналогов | **knowledge(search)** — ChromaDB + GigaChat Embeddings. |
| 4. Суммаризация | Саммари звонков, отчёты по тикетам, executive-summary | Модель суммирует на основе контекста (в т.ч. результатов RAG); отдельного tool нет. |
| 5. Генерация текстов | Ответы клиентам, карточки товара, инструкции | Ответы в диалоге; file(write) для документов; при наличии — генерация DOCX. |
| 6. Перефразирование / упрощение / локализация | «Сложный ответ юриста» → «для клиента» / «для вики» | Возможности модели в диалоге; не зашито в отдельный tool. |
| 7. Нормализация / обогащение данных | Сопоставление SKU, мэппинг категорий, коды причин возврата | В демо не выделено; в клиенте — сценарий поверх данных (возможно свой tool/интеграция). |
| 8. Планирование и разбиение на шаги | Разложить запрос в план: что сделать, роли, зависимости | **tasks(add, list, update)**; cron (напоминания); модель может предлагать план в ответе. |
| 9. Функциональные вызовы (инструменты/API) | «Прочитал → узнал → сделал»: вопрос → данные из ERP → ответ | **knowledge(search)**, **web(search/read_url)**, **file**, **project**, **tasks**, **cron** — типичный цепочки вызовов. |
| 10. Табличные задачи | Ad-hoc аналитика, подсказки формул, text-to-SQL | В GigaBot — чтение XLSX, file(read); полноценный text-to-SQL — сценарий для корпоративного контура (GigaEnterprise). |

**Вывод:** примитивы 3, 8, 9 в GigaBot представлены явно (knowledge, tasks, cron, web, file, project); остальные — через возможности модели и комбинации tools. Для CSM слайд «примитивы LLM» можно подкрепить примерами из демо (поиск по базе, задачи, вызов нескольких инструментов под один запрос).

## 10 признаков плохого кейса — GigaChat не «серебряная пуля»

| № | Признак | Корректность | Связь с ограничениями бота / LLM |
|---|---------|--------------|-----------------------------------|
| 1 | 100% детерминированность: бухпроводки, налоговые расчёты, дозировки, управление оборудованием | Корректно | LLM не гарантирует детерминизм; для таких кейсов — правила и проверки, не «свободная» генерация. |
| 2 | Полномочия на действия: публичный бот создаёт заказы/переводы/тикеты | Корректно | В демо бот не выполняет финансовые/юридические действия без контроля; у клиента — разграничение полномочий и подтверждения. |
| 3 | Нет доступа к источнику истины: не определены критерии «хорошего» результата | Корректно | RAG как раз даёт «источник истины» (документы); без него ответы могут быть некорректны. |
| 4 | Редкие задачи: редко возникают, каждый раз новое решение | Корректно | LLM сильнее там, где есть паттерны и данные (RAG, типовые задачи); разовые уникальные кейсы — слабее. |
| 5 | Чистая математика / оптимизация / алгоритм: биллинг, роутинг, оптимизация | Корректно | Задача для специализированных систем, не для генеративной модели. |
| 6 | Анализ временных рядов: скоринг, режимы оборудования | Корректно | Не примитив текущего демо; у клиента — отдельные ML-модули. |
| 7 | Нет метрики процесса: нельзя замерить «как было» и «как стало» | Корректно | Для внедрения нужны метрики (см. документ 03); иначе ценность не доказать. |
| 8 | Нет базовой автоматизации: нечего улучшать | Корректно | Агент улучшает существующий процесс (документы, задачи, диалог); без процесса внедрять некуда. |
| 9 | Невозможно логирование: нельзя записать действия пользователей/операторов | Корректно | В GigaBot есть история диалога, MEMORY.md, HISTORY.md; у клиента логирование обязательно. |
| 10 | Нет владельца процесса или ответственного за метрику | Корректно | Внедрение и метрики должны иметь владельца на стороне клиента. |

**Вывод:** слайд «10 признаков плохого кейса» корректен и хорошо стыкуется с ограничениями LLM и с тем, что GigaBot демонстрирует (есть RAG, логирование, задачи, владелец — пользователь/презентатор). Для CSM полезно кратко озвучить: «GigaChat не серебряная пуля — вот когда не тянет, а вот когда тянет» (хорошие признаки + примитивы).
