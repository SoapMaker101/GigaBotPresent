### Пояснения по каждому признаку (хороший кейс)

**Признак 1. Объём и рутина**
- **Что:** Много повторяющихся операций: чтение и написание текста, сравнение форм, тикетов, документов.
- **Где:** Бек-офисы, поддержка, экспедиция документов, закупки — везде, где сотрудники каждый раз делают однотипные действия с текстом.
- **Как в GigaBot:** RAG даёт поиск по документам без ручного листания; задачи и напоминания снимают рутину учёта; индексация папок и ответы по базе знаний — как раз массовые повторяемые сценарии.
- **Почему хороший кейс:** LLM и RAG автоматизируют рутину и дают измеримое сокращение времени и высвобождение FTE.

**Признак 2. Разнобой форматов**
- **Что:** Данные приходят в разных форматах: PDF, сканы, фото, полуструктурированные файлы, данные «на полях».
- **Где:** Документооборот, входящая корреспонденция, заявки с приложениями, отчёты из разных систем.
- **Как в GigaBot:** Поддержка PDF, DOCX, XLSX, TXT; OCR по фото; в RAG — единый семантический поиск поверх форматов, пользователь не думает «в каком формате искать».
- **Почему хороший кейс:** Один интерфейс запроса вместо ручной работы с каждым форматом отдельно; клиент получает единую точку доступа к разрозненным данным.

**Признак 3. Разрыв «человек–система»**
- **Что:** Сотрудникам приходится глазами читать документы или экраны, сопоставлять данные, затем вручную вбивать их в CRM/ERP.
- **Где:** Ввод заявок, перенос данных из писем в тикеты, заполнение форм на основе отсканированных документов.
- **Как в GigaBot:** Агент сам читает документы, ищет в RAG и формирует ответы; диалог даёт «запрос → ответ» без ручного перекладывания в систему (в демо это показано на примере базы знаний и задач).
- **Почему хороший кейс:** Сокращается разрыв между «узнал» и «зафиксировал в системе» — меньше ошибок, быстрее цикл обработки.

**Признак 4. Исключения и «длинный хвост»**
- **Что:** Много нестандартных кейсов, которые не покрыты жёсткими правилами и скриптами.
- **Где:** Поддержка с необычными запросами, разовые запросы к регламентам, сложные формулировки клиентов.
- **Как в GigaBot:** LLM обрабатывает свободные формулировки; RAG подтягивает релевантные фрагменты при нестандартных вопросах, без необходимости заранее прописывать все варианты.
- **Почему хороший кейс:** Гибкость LLM как раз для длинного хвоста; жёсткая автоматизация здесь не масштабируется, а человек+LLM — да.

**Признак 5. Мультиканальность**
- **Что:** Обращения приходят по разным каналам: e-mail, чат, звонки, мессенджеры, соцсети.
- **Где:** Контакт-центры, единая платформа поддержки, омниканальный сервис.
- **Как в GigaBot:** В демо — один канал (Telegram); мультиканальность — это уже архитектура GigaEnterprise (несколько каналов в одном агенте). Для слайда признак корректен как цель клиента.
- **Почему хороший кейс:** Один агент поверх каналов даёт единую логику и метрики вместо разрозненных скриптов по каждому каналу.

**Признак 6. Мультиязычность**
- **Что:** Стык данных на нескольких языках: контракты, ВЭД, внутренние регламенты на разных языках.
- **Где:** Внешнеэкономическая деятельность, международные проекты, мультиязычные базы знаний.
- **Как в GigaBot:** GigaChat и RAG в демо ориентированы на русский; мультиязычность — общий признак хорошего кейса для LLM, не специфика текущего демо.
- **Почему хороший кейс:** LLM умеет работать с несколькими языками в одном контуре; для клиента это снятие барьера «разные языки — разные системы».

**Признак 7. SLA / время-до-ответа важно**
- **Что:** Критична скорость ответа или обработки: контакт-центры, закупки, поддержка, риск-оценка.
- **Где:** КЦ, сервисные дескты, процессы с жёсткими SLA, операционные комитеты.
- **Как в GigaBot:** Быстрый ответ за счёт RAG (семантический поиск) и агента (один запрос — цепочка вызовов); для CSM это операционная метрика «сокращение времени процессов».
- **Почему хороший кейс:** Автоматизация ответов и поиска напрямую влияет на время реакции и выполнение SLA — понятная ценность для клиента.

**Признак 8. Наличие «центров обслуживания»**
- **Что:** Есть структурированные центры: контакт-центры, бек-офисы, экспедиции документов.
- **Где:** КЦ, офисы обработки заявок, подразделения с типовыми процессами и объёмом обращений.
- **Как в GigaBot:** Демо — прототип такого центра: один агент с базой знаний и задачами; у клиента тот же принцип, масштаб до полноценного КЦ/бек-офиса.
- **Почему хороший кейс:** Агент с RAG и задачами естественно ложится на организационную структуру «центра» и даёт измеримый эффект на FTE и качество.

**Признак 9. Текучесть персонала**
- **Что:** Много знаний живёт «в головах», внутренние регламенты разбросаны, новые сотрудники долго входят в курс.
- **Где:** Любые подразделения с высокой текучкой или сложной экспертизой; онбординг, инструкции, эскалации.
- **Как в GigaBot:** База знаний (RAG) фиксирует знания в документах; поиск по ним не зависит от конкретного сотрудника — ответ даёт система, а не человек.
- **Почему хороший кейс:** Знания переходят из «голов» в контур с LLM; снижаются риски при уходе людей и ускоряется онбординг.

**Признак 10. Сезонные пики / «дни X»**
- **Что:** Всплески обращений или документооборота в определённые периоды или даты.
- **Где:** Отчётные кампании, распродажи, конец квартала, массовые рассылки и ответы на них.
- **Как в GigaBot:** Агент масштабируется без пропорционального роста числа операторов; для клиента — аргумент про avoided costs и отказ от набора FTE под пики.
- **Почему хороший кейс:** LLM-агент снимает пиковую нагрузку без линейного роста штата — понятная финансовая и операционная выгода.

## Примитивы LLM — маппинг на инструменты GigaBot

| Примитив | Описание | Инструмент / сценарий в GigaBot |
|----------|----------|----------------------------------|
| 1. Классификация | Маршрутизация тикетов, темы обращений, intent, риск-метки | Модель по тексту запроса выбирает действие (RAG, задачи, файлы и т.д.); явного отдельного tool нет. |
| 2. Извлечение сущностей | Заявки, анкеты, счета, контракты, e-mail-заказы | OCR (текст с фото); чтение файлов (file read); при индексации в RAG — текст из PDF/DOCX. |
| 3. Семантический поиск | База знаний, регламенты, каталоги, подбор аналогов | **knowledge(search)** — ChromaDB + GigaChat Embeddings. |
| 4. Суммаризация | Саммари звонков, отчёты по тикетам, executive-summary | Модель суммирует на основе контекста (в т.ч. результатов RAG); отдельного tool нет. |
| 5. Генерация текстов | Ответы клиентам, карточки товара, инструкции | Ответы в диалоге; file(write) для документов; при наличии — генерация DOCX. |
| 6. Перефразирование / упрощение / локализация | «Сложный ответ юриста» → «для клиента» / «для вики» | Возможности модели в диалоге; не зашито в отдельный tool. |
| 7. Нормализация / обогащение данных | Сопоставление SKU, мэппинг категорий, коды причин возврата | В демо не выделено; в клиенте — сценарий поверх данных (возможно свой tool/интеграция). |
| 8. Планирование и разбиение на шаги | Разложить запрос в план: что сделать, роли, зависимости | **tasks(add, list, update)**; cron (напоминания); модель может предлагать план в ответе. |
| 9. Функциональные вызовы (инструменты/API) | «Прочитал → узнал → сделал»: вопрос → данные из ERP → ответ | **knowledge(search)**, **web(search/read_url)**, **file**, **project**, **tasks**, **cron** — типичный цепочки вызовов. |
| 10. Табличные задачи | Ad-hoc аналитика, подсказки формул, text-to-SQL | В GigaBot — чтение XLSX, file(read); полноценный text-to-SQL — сценарий для корпоративного контура (GigaEnterprise). |

**Вывод:** примитивы 3, 8, 9 в GigaBot представлены явно (knowledge, tasks, cron, web, file, project); остальные — через возможности модели и комбинации tools. Для CSM слайд «примитивы LLM» можно подкрепить примерами из демо (поиск по базе, задачи, вызов нескольких инструментов под один запрос).

## 10 признаков плохого кейса — GigaChat не «серебряная пуля»

| № | Признак | Корректность | Связь с ограничениями бота / LLM |
|---|---------|--------------|-----------------------------------|
| 1 | 100% детерминированность: бухпроводки, налоговые расчёты, дозировки, управление оборудованием | Корректно | LLM не гарантирует детерминизм; для таких кейсов — правила и проверки, не «свободная» генерация. |
| 2 | Полномочия на действия: публичный бот создаёт заказы/переводы/тикеты | Корректно | В демо бот не выполняет финансовые/юридические действия без контроля; у клиента — разграничение полномочий и подтверждения. |
| 3 | Нет доступа к источнику истины: не определены критерии «хорошего» результата | Корректно | RAG как раз даёт «источник истины» (документы); без него ответы могут быть некорректны. |
| 4 | Редкие задачи: редко возникают, каждый раз новое решение | Корректно | LLM сильнее там, где есть паттерны и данные (RAG, типовые задачи); разовые уникальные кейсы — слабее. |
| 5 | Чистая математика / оптимизация / алгоритм: биллинг, роутинг, оптимизация | Корректно | Задача для специализированных систем, не для генеративной модели. |
| 6 | Анализ временных рядов: скоринг, режимы оборудования | Корректно | Не примитив текущего демо; у клиента — отдельные ML-модули. |
| 7 | Нет метрики процесса: нельзя замерить «как было» и «как стало» | Корректно | Для внедрения нужны метрики (см. документ 03); иначе ценность не доказать. |
| 8 | Нет базовой автоматизации: нечего улучшать | Корректно | Агент улучшает существующий процесс (документы, задачи, диалог); без процесса внедрять некуда. |
| 9 | Невозможно логирование: нельзя записать действия пользователей/операторов | Корректно | В GigaBot есть история диалога, MEMORY.md, HISTORY.md; у клиента логирование обязательно. |
| 10 | Нет владельца процесса или ответственного за метрику | Корректно | Внедрение и метрики должны иметь владельца на стороне клиента. |

### Пояснения по каждому признаку (плохой кейс)

**Признак 1. 100% детерминированность**
- **Что:** От результата требуется строгая предсказуемость: одна и та же входная данные всегда дают один и тот же выход (бухпроводки, налоги, дозировки, управление оборудованием).
- **Где:** Бухгалтерия, регуляторика, медицина (дозировки), промышленная автоматизация.
- **Почему не тянет LLM:** Генеративные модели не гарантируют детерминизм; ответ может слегка варьироваться. Для таких кейсов нужны правила, формулы и проверки, а не свободная генерация.
- **Почему важно:** CSM не должен обещать «бот посчитает налог» без жёсткой логики и контроля; иначе — репутационные и регуляторные риски.

**Признак 2. Полномочия на действия**
- **Что:** Бот или система от имени клиента совершают необратимые действия: заказы, переводы, создание тикетов с последствиями.
- **Где:** Публичные чат-боты с доступом к платежам, массовая выдача справок, автоматическое закрытие заявок.
- **Почему не тянет LLM:** В демо бот не выполняет финансовые/юридические действия без контроля; у клиента обязательны разграничение полномочий, подтверждения и аудит.
- **Почему важно:** Озвучивать клиенту: агент помогает готовить решение, но критические действия — только с санкции человека и прозрачным логированием.

**Признак 3. Нет доступа к источнику истины**
- **Что:** Не определено, по каким данным или критериям считать результат «правильным»; нет эталона для проверки ответа.
- **Где:** Запросы «как угодно ответь», сценарии без регламентов и документов, домыслы вместо фактов.
- **Почему не тянет LLM:** Без RAG или чётких данных модель может галлюцинировать; RAG как раз даёт источник истины (документы). Без него внедрение рискованно.
- **Почему важно:** Для любого кейса с LLM нужно формулировать: откуда берётся истина (документы, API, регламенты) и как проверяется ответ.

**Признак 4. Редкие задачи**
- **Что:** Задачи возникают редко, каждый раз уникальны, нет накопленных паттернов и данных для обучения или поиска.
- **Где:** Разовые расследования, уникальные запросы, задачи без повторяемости.
- **Почему не тянет LLM:** LLM сильнее там, где есть паттерны, RAG и типовые сценарии; разовые уникальные кейсы дают мало выгоды от автоматизации и выше риск ошибки.
- **Почему важно:** Фокус на процессах с повторяемостью и объёмом; разовые кейсы лучше оставлять экспертам без обещаний «бот всё решит».

**Признак 5. Чистая математика / оптимизация / алгоритм**
- **Что:** Задача сводится к точным вычислениям, оптимизации, биллингу, роутингу — без естественного языка и неоднозначности.
- **Где:** Биллинг, маршрутизация, расчёт тарифов, численная оптимизация.
- **Почему не тянет LLM:** Это зона специализированных систем и алгоритмов, не генеративной модели; LLM не замена калькулятору и солверу.
- **Почему важно:** Не подменять классическую автоматизацию и расчёты «умным чатом»; клиенту объяснять границы LLM.

**Признак 6. Анализ временных рядов**
- **Что:** Нужны скоринг, прогнозы, режимы оборудования по временным рядам и историческим данным.
- **Где:** Риск-аналитика, предиктивное обслуживание, прогнозная аналитика.
- **Почему не тянет LLM:** Это не примитив текущего демо; у клиента решаются отдельными ML-модулями (модели по рядам, а не генеративный текст).
- **Почему важно:** Не путать «LLM для текста и диалога» с «ML для рядов и скоринга»; для последнего — свои продукты и интеграции.

**Признак 7. Нет метрики процесса**
- **Что:** Нельзя замерить «как было» и «как стало» — нет базовых показателей процесса до и после внедрения.
- **Где:** Хаотичные процессы без учёта времени, объёма, ошибок; пилоты «просто попробуем».
- **Почему не тянет LLM:** Без метрик ценность внедрения не доказать; документ 03 как раз про финансовые, операционные и стратегические метрики.
- **Почему важно:** Перед внедрением договариваться с клиентом о метриках; иначе невозможно обосновать ROI и продление проекта.

**Признак 8. Нет базовой автоматизации**
- **Что:** Процесс полностью ручной и неформализованный; нечего «улучшать» — нет ни документов, ни правил, ни учёта.
- **Где:** Полный хаос в процессах, отсутствие регламентов и баз данных.
- **Почему не тянет LLM:** Агент улучшает существующий процесс (документы, задачи, диалог); без структуры и данных внедрять некуда — сначала базовая упорядоченность.
- **Почему важно:** LLM — следующий шаг после того, как есть хотя бы документы, задачи или учёт; иначе клиенту сначала помогать с основой.

**Признак 9. Невозможно логирование**
- **Что:** Нельзя записать, кто что спросил, какой ответ был дан, какие действия выполнены — нет аудита и контроля.
- **Где:** Режимы «без логов», несоответствие требованиям по хранению и расследованию инцидентов.
- **Почему не тянет LLM:** В GigaBot есть история диалога, MEMORY.md, HISTORY.md; у клиента логирование обязательно для комплаенса и разбора претензий.
- **Почему важно:** Обещать только такие внедрения, где действия и ответы логируются и доступны владельцу процесса и контролёрам.

**Признак 10. Нет владельца процесса или ответственного за метрику**
- **Что:** Никто на стороне клиента не отвечает за результат внедрения и за метрики; нет заказчика и приёмки.
- **Где:** Инициативы «снизу» без бюджета и KPI, пилоты без ответственного за продление.
- **Почему не тянет LLM:** Внедрение и метрики должны иметь владельца на стороне клиента; иначе пилот «повиснет» и ценность не будет зафиксирована.
- **Почему важно:** Перед стартом определять владельца процесса и ответственного за метрику; без этого не начинать или явно ограничивать пилот по срокам и целям.
