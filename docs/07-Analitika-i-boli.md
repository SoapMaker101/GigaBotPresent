# Аналитика диалогов и боли

**Для презентации CSM в Сбере.** Документ уточняет, что считать «аналитикой диалогов» в контексте GigaBot, и как текущий функционал и доработки по ТЗ закрывают боли (мониторинг задач, навигация по RAG).

---

## Что считать «аналитикой диалогов» в контексте бота

В рамках презентации под «аналитикой диалогов» понимается не полная BI-система, а **обозримая картина того, что происходит в контуре**:

1. **Что загружено**
   - Проекты: список папок в `workspace/projects/`, их структура (подпапки: Договоры, Документация, Сметы, Фото, Переписка).
   - Файлы: какие файлы в каких проектах и папках лежат (через `project(list)` или обход файловой системы в сводке).

2. **Какие задачи**
   - Источник: `tasks/tasks.json` (или аналог в форке).
   - Содержимое: список задач с полями title, project, status, priority, deadline, created_at.
   - Аналитика: сколько задач в работе, сколько с дедлайном на эту неделю, по каким проектам распределены.

3. **Что в базах знаний (RAG)**
   - Источник: ChromaDB (коллекции), вызов `knowledge(list_projects)`.
   - Содержимое: имена проектов RAG, количество проиндексированных документов/фрагментов в каждой коллекции.
   - Аналитика: какие базы есть, в каком объёме проиндексированы; при необходимости — когда последний раз индексировали (если это сохраняется).

4. **История и память**
   - `memory/HISTORY.md` — журнал событий (append-only), можно искать по тексту (grep).
   - `memory/MEMORY.md` — долговременная память агента (важные факты).
   - Сессионная история диалога хранится в SessionManager (в рамках сессии); при консолидации часть уходит в HISTORY/MEMORY.

Для презентации достаточно показать **сводный обзор**: «что загружено, какие задачи, что в базах знаний» — одной командой или одним экраном (реализация в форке по ТЗ — см. tech-spec).

---

## Боль 1: Мониторинг задач

**Суть:** Много задач по разным проектам и срокам; сложно держать в голове приоритеты и просрочки.

**Как закрывает текущий функционал:**

- **tasks(list)** — список всех задач с фильтрами по `project` и `status`; в выводе отображаются приоритет (high/medium/low) и дедлайн.
- **tasks(list)** + запрос вида «подготовь отчёт по приоритету и дедлайну» — агент интерпретирует список и формирует текстовый отчёт (что первым, что позже).

**Как закрывают доработки по ТЗ:**

- Введение полей «сложность» и/или «критичность» в модель задачи (опционально) и явный отчёт по факторам критичность/сложность/дедлайн.
- Сводка аналитики (один вызов или один экран): в блоке «задачи» вывести краткую статистику (в работе / с дедлайном на неделю / по проектам).

Итог: мониторинг задач закрывается комбинацией существующего `tasks` и, при необходимости, отчёта/сводки из ТЗ.

## Боль 2: Навигация по RAG («не всегда помню, где что лежит»)

**Суть:** После индексации многих документов и проектов непонятно, в какой базе знаний что искать.

**Как закрывает текущий функционал:**

- **knowledge(list_projects)** — возвращает список проектов RAG с отображаемыми именами и количеством документов (фрагментов) в каждой коллекции. Пользователь или агент видит, какие базы есть.
- Поиск по конкретному проекту: **knowledge(search, project="...", query="...")**. При неочевидном выборе проекта пользователь может спросить «найди во всех базах про X» — агент может последовательно вызвать search по нескольким проектам (если это заложено в сценарий) или сначала list_projects и затем поиск по одному/нескольким.

**Как закрывают доработки по ТЗ:**

- Сводка аналитики: в блоке «базы знаний» вывести список проектов RAG и объёмы (уже даёт knowledge(list_projects)); при необходимости добавить краткое описание или дату последней индексации, если это будет реализовано в форке.

Итог: навигация по RAG закрывается списком проектов и семантическим поиском; сводка в одном месте усиливает обзорность.

## Сводная таблица

| Боль | Текущий функционал | Доработки по ТЗ |
|------|--------------------|-----------------|
| Мониторинг задач | tasks(list), приоритет и дедлайн в выводе; запрос отчёта агенту | Отчёт по критичности/сложности/дедлайну; блок «задачи» в сводке аналитики |
| Навигация по RAG | knowledge(list_projects), knowledge(search) по проекту | Блок «базы знаний» в сводке аналитики (список + объёмы) |

Обе боли в презентации можно озвучить как «мои проблемы», а затем показать, как контур с LLM (список задач, список баз, поиск по запросу и при необходимости сводка) их снимает.
