# Тема 1: Внедрение LLM в контур

**Для презентации CSM в Сбере.** Цель блока — показать, как большая языковая модель (LLM) встраивается в инфраструктурный контур и закрывает операционные боли.

---

## Аналогия контура: инфраструктура как Cloud.ru

В презентации контур инфраструктуры можно описывать по аналогии с **Cloud.ru** (или SberCloud):

- **Инфраструктура** — где развёрнуто приложение: серверы, среда выполнения, хранилища, сеть.
- **LLM в контуре** — GigaChat подключён как сервис к этому контуру: один запрос от пользователя → обработка в приложении → вызов GigaChat (чат + при необходимости эмбеддинги) → выполнение инструментов (файлы, базы знаний, задачи) → ответ пользователю.

GigaBot в данной постановке — **пример такого контура**: приложение работает «где-то» (в т.ч. может быть развёрнуто на Cloud.ru), к нему подключён GigaChat. Клиент видит один интерфейс (например Telegram), а «под капотом» — полноценное внедрение LLM: не просто чат, а агент с инструментами и данными.

Для CSM важно: **внедрение LLM в контур** = инфраструктура (Cloud.ru) + модель (GigaChat) + бизнес-логика и инструменты (проекты, базы знаний, задачи, напоминания). Это и есть «как происходит внедрение ИИ в LLM» на живом примере.

---

## «Мои» бизнес-процессы в контуре

В рамках демо можно описать условные бизнес-процессы, которые живут в этом контуре:

1. **Постановка задач и создание баз знаний с агентной системой**
   - Пользователь в диалоге ставит задачи (с дедлайнами, приоритетами), создаёт проекты и базы знаний, индексирует документы.
   - Агент (LLM + инструменты) выполняет действия: создание проектов, индексация папок/файлов, поиск по RAG, добавление и обновление задач, напоминания по расписанию.

2. **Аналитика диалогов с ботом**
   - Что было загружено: в какие проекты и папки попали файлы.
   - Какие задачи созданы и в каком статусе (список задач, приоритет, дедлайн).
   - Что можно сказать про базу знаний: какие проекты RAG есть, что в них проиндексировано, сколько фрагментов.

Эти «процессы» показывают, что LLM в контуре не просто отвечает на вопросы, а **управляет структурированными данными** (проекты, файлы, RAG, задачи) через единый диалог.

## Боли, которые закрывает контур с LLM

Для истории на презентации удобно сформулировать две боли:

1. **Мониторинг задач**
   - Много задач по разным проектам и срокам; сложно держать в голове, что делать первым, что просрочено, что уже сделано.
   - В контуре с LLM: единый список задач (tool `tasks`), фильтрация по проекту/статусу, приоритет и дедлайн; по желанию — отчёт по критичности, сложности и дедлайну (см. документ 02 и ТЗ форка).

2. **Работа с базами RAG («не всегда помню, где что лежит»)**
   - Документы разбросаны по проектам и папкам; после индексации неочевидно, в какой базе знаний что искать.
   - В контуре с LLM: список проектов RAG (`knowledge(list_projects)`), семантический поиск по запросу без необходимости помнить точные пути; агент сам выбирает проект или ищет по нескольким базам.

Итог для CSM: **внедрение LLM в контур** даёт не «просто чат», а систему, которая закрывает конкретные операционные боли через диалог, инструменты и данные в единой среде (аналогия — Cloud.ru + GigaChat + агентная оболочка).
